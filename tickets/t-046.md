# T-046: Dual-team scoring — fix half-inning score attribution + opponent scoring modes

## Status
Done

## Epic
E-02 Core Entry Flows (bugfix + UX extension)

## Priority
P0

## Goal
Fix a critical scoring bug where runs are incorrectly accumulated regardless of half-inning, causing
the home team to be credited for runs scored while the opponent is batting. Add support for two
opponent-scoring modes: (A) **full lineup mode** — the opponent has a registered roster/lineup and
is scored exactly like the home team via a side-switch mechanism; (B) **quick-score mode** — a
simple +/– run counter for the opponent half-inning when no away roster is loaded.

## Why it matters
As reported during live testing: when the away team bats, the UI still shows the home lineup and
credits any recorded run to the home team. A scorekeeper recording a full game will produce
completely incorrect box scores. This is a P0 correctness bug. The two scoring modes address the
real-world split: some users will have both rosters loaded, others will only have their own team's
roster.

## Bug details
**Root cause in `useGameState.ts` lines 19–21:**
```ts
for (const ev of events) {
  ourRuns += ev.runsScored   // ← credits ALL runs to "us", ignoring halfInning
```
`ev.halfInning` is stored on every `PlayEventRow` but is never consulted when summing runs.
The fix is to only add `ev.runsScored` to `ourRuns` when `ev.halfInning` matches the half
where OUR team bats (home team bats bottom; away team bats top).

**Secondary issue:** The live scoring screen has no concept of "opponent half-inning" — it always
shows our lineup and offers the full outcome grid even when it is the opponent's turn to bat.

## Scope

### Phase 1 — Core bug fix (score attribution)
- Fix `buildLiveState` in `useGameState.ts` to only credit `ourRuns` during OUR half-innings.
- Add `opponentRuns` accumulation from the OTHER half-innings' `runsScored`.
- Pass `opponentRuns` into `LiveGameState` and wire into `ScoreboardHUD`.

### Phase 2 — Half-inning mode switching
- Add `scoringMode` field to `LiveGameState`:
  - `'our_half'` — we are batting; show our lineup + outcome grid (current behavior)
  - `'opponent_quick'` — opponent is batting, no roster; show simple +/– run counter
  - `'opponent_lineup'` — opponent is batting, roster loaded; show their lineup + outcome grid
- When the outs state machine crosses 3 outs, auto-switch mode:
  - top → bottom: if `homeAway === 'away'`, we just finished batting → switch to `opponent_*`
  - bottom → top (inning over): if `homeAway === 'home'`, we just finished batting → switch
- Add a "End Half-Inning" button / "Switch sides" affordance the user can also tap manually.

### Phase 3 — Quick-score mode UI (`opponent_quick`)
- Replace the outcome grid with a minimal panel:
  - Header: "▲/▼ [Inning] — [Opponent] at bat"
  - Large +/– stepper for opponent runs this half-inning (does NOT create `play_event` rows;
    instead writes an `opponent_runs` delta directly — see data model note below)
  - "End Half-Inning" button to manually advance to the next half
  - Undo button to decrement opponent runs

### Phase 4 — Full lineup mode UI (`opponent_lineup`)
- Allow an opponent team to be linked at game-setup time (optional — select from existing teams or
  create a new opposing team stub on the spot).
- At `opponent_lineup` halves: show the opponent's lineup instead of ours and record plays
  identically to Phase 1, with `halfInning` set to the opponent's half.
- Stat rebuild for opponent play events DOES NOT roll into our `player_game_stats` —
  those events are owned by the opponent's `teamId`.
- The `GameRow` needs an optional `opponentTeamId` field (nullable, schema migration required).

### Phase 5 — Game setup update
- Add optional "Select opponent roster" step to `GameSetupScreen` — pick an existing team or skip.
  - If an opponent team is selected: `GameRow.opponentTeamId` is set → `opponent_lineup` mode used
    during their halves.
  - If skipped: `GameRow.opponentTeamId` is null → `opponent_quick` mode used.
- Opponent team selection can also be added/changed from the live scoring screen for flexibility.

## Data model changes

### `GameRow` — add optional field
```ts
opponentTeamId?: string   // null/undefined = quick-score mode; set = full lineup mode
```
Dexie migration: add `opponentTeamId` to the `games` store definition (non-indexed is fine).

### Opponent run storage — quick-score mode
Quick-score opponent runs need a home in the DB so they survive a reload. Options:
- **Option A (preferred):** Store a synthetic `play_event` with `outcome: 'field_out'` (or a new
  `'opponent_run'` marker) + `runsScored: N` and `batterPlayerId: ''` for opponent quick halves.
  This keeps the event stream the single source of truth and the score rebuilder already handles
  `runsScored` summed per halfInning.
- **Option B:** Add an `opponentScoreByInning: number[]` JSON field to `GameRow`. Simpler but
  splits source of truth.

**Decision: Use Option A.** Store opponent quick-score runs as a special play event with
`outcome: 'opponent_score'` (new `PlayOutcome` union member), `batterPlayerId: ''`, and
`runsScored` = the entered count. The stat rebuilder ignores events with `outcome === 'opponent_score'`
when computing player stats. `buildLiveState` reads them to populate `opponentRuns`.

> ⚠️ Adding `'opponent_score'` to the `PlayOutcome` union is a schema-touching change.
> Claim the `data/repositories/` and `domain/stat-engine/` shared-file ownership before starting.

## Implementation notes
- The `halfInning` that "our team" bats:
  - `homeAway === 'home'` → we bat in `'bottom'` halves
  - `homeAway === 'away'` → we bat in `'top'` halves
- `buildLiveState` scoring fix (single expression):
  ```ts
  const ourHalf: HalfInning = game.homeAway === 'home' ? 'bottom' : 'top'
  // inside the event loop:
  if (ev.halfInning === ourHalf) ourRuns += ev.runsScored
  else opponentRuns += ev.runsScored
  ```
- Mode auto-switch logic belongs in `buildLiveState` (derived from current `halfInning` after
  replaying all events), not in a separate effect.
- The scoreboard HUD already accepts `homeScore`/`awayScore` separately — the fix threads
  the correct values in; no HUD layout change needed.
- Quick-score panel must be full-width, large touch targets (≥ 44px), and use the Clay Orange
  palette to visually signal "opponent territory."
- `'opponent_score'` events must be filtered out of OUTCOME_LABELS display in the play log
  (or given a label like "Opponent Runs (manual)").
- Dexie migration must be a forward-only `version(N+1).stores({ games: '...' })` block — never
  drop existing indexes.
