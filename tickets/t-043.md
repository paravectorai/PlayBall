## Goal
Audit and fix the app's bundle size, Dexie IndexedDB query plans, and render hot paths so the app meets its load-time and auto-save latency targets on mid-range phones.

## Why it matters
T-040/T-041 hardened the PWA install experience. Before release (T-045), we need confidence that the app's runtime performance holds up at the data volumes a real season produces — ~15 players, ~20 games, ~50 events per game.

## Scope
- Measure gzip bundle size; document whether code splitting is warranted
- Add compound Dexie indexes where queries currently use JS-side `.sortBy()` after a `where()` filter
- Fix `nextSequenceNumber` to avoid loading all game events on every play save
- Replace O(n×m) `players.find()` in the batting order render with an O(1) Map lookup
- Document all findings and fixes in completion notes

## Out of scope
- Dexie live query subscriptions
- Virtual list / windowing for large datasets
- Full Lighthouse CI integration
- Any new feature UI

## Implementation notes
- Schema version bump from 1 → 2 adds indexes only; no data migration needed
- Dexie handles index-only migrations automatically (no upgrade callback required)
- Compound indexes follow Dexie syntax: `[a+b]` in `.stores()` string
- Use `Dexie.minKey` / `Dexie.maxKey` with `.between()` to query compound indexes for partial key ranges
- `.reverse().toArray()` on a compound index gives native descending cursor order

## Acceptance criteria
- `nextSequenceNumber` uses `.last()` on a compound `[gameId+sequenceNumber]` index, not a full load + JS sort
- `findByGameOrdered` uses the same compound index for native ordering
- `GameRepository.findBySeason` uses a compound `[seasonId+gameDate]` index
- `LineupRepository.findByGame` uses a compound `[gameId+battingOrder]` index
- Batting order render in `LiveScoringScreen` uses a `playerMap` Map for O(1) lookup
- DB schema is at version 2; existing data survives the migration
- `npm run build` succeeds; gzip JS ≤ 250KB; all existing tests pass

## Completion notes
Completed 2026-06-11.

**Bundle size audit:**
- JS: 208.20KB raw / 60.28KB gzip — well under target. Single chunk is appropriate for an offline-first PWA where the service worker caches all assets; route-based splitting would add round-trips for no benefit.
- CSS: 17.57KB raw / 4.05KB gzip.

**Dexie query fixes:**
- Added `[gameId+sequenceNumber]` compound index to `playEvents` and `[seasonId+gameDate]` to `games` and `[gameId+battingOrder]` to `lineups` — schema bumped to version 2.
- `PlayEventRepository.findByGameOrdered`: now uses compound index `.between()` + `.toArray()` — eliminates JS-side sort on every live-scoring state rebuild.
- `PlayEventRepository.nextSequenceNumber`: now uses compound index + `.last()` — eliminates loading all game events on every play save.
- `GameRepository.findBySeason`: now uses compound index `.between().reverse().toArray()` — eliminates JS-side sort.
- `LineupRepository.findByGame`: now uses `[gameId+battingOrder]` compound index.

**Render fix:**
- `LiveScoringScreen` batting order list: replaced per-row `players.find()` (O(n×m)) with a `playerMap` Map built once per render (O(n+m)).

**Validation:**
- `npm test`: 97/97 pass.
- `npm run build`: clean; gzip JS 60.28KB.
